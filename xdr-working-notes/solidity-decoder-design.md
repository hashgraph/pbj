# Solidity XDR Decoder — Output Format Design

This document specifies the Solidity contracts PBJ will generate for decoding XDR-encoded
calldata on the EVM. It covers the target output format, library vs contract structure,
calldata pointer arithmetic patterns, gas cost analysis, and rejected alternatives.

For the XDR wire format used as input, see [xdr-design.md](../xdr/xdr-design.md) §2–§4.
For the Java codec design, see [codec-xdr.md](../codec-xdr.md).

## 1. Target Solidity Output — Worked Example

### Message: `ClprQueueMetadata`

```protobuf
message ClprQueueMetadata {
    uint64 next_message_id     = 1;  // XDR: 4B presence + 8B hyper = 12 bytes
    uint64 acked_message_id    = 2;  // XDR: 4B presence + 8B hyper = 12 bytes
    bytes  sent_running_hash   = 3;  // XDR: 4B presence + 32B opaque[32] = 36 bytes
    bytes  received_running_hash = 4; // XDR: 4B presence + 32B opaque[32] = 36 bytes
    uint64 received_message_id = 5;  // XDR: 4B presence + 8B hyper = 12 bytes
}
```

### XDR Wire Layout

All fields are singular with non-default values, so each is written as presence=1 followed
by the value. Fixed-length `bytes[32]` fields use XDR `opaque[32]` — no length prefix, just
32 raw bytes (already 4-byte aligned so no padding).

```
Offset  Size  Content
------  ----  -------
     0     4  presence flag for next_message_id (0x00000001)
     4     8  next_message_id value (uint64 big-endian)
    12     4  presence flag for acked_message_id (0x00000001)
    16     8  acked_message_id value (uint64 big-endian)
    24     4  presence flag for sent_running_hash (0x00000001)
    28    32  sent_running_hash value (32 raw bytes, no length prefix)
    60     4  presence flag for received_running_hash (0x00000001)
    64    32  received_running_hash value (32 raw bytes, no length prefix)
    96     4  presence flag for received_message_id (0x00000001)
   100     8  received_message_id value (uint64 big-endian)
------  ----
   108        total bytes (all fields present)
```

All offsets are compile-time constants because every field is fixed-size. This is the ideal
case for the EVM decoder.

### Generated Solidity Library

```solidity
// SPDX-License-Identifier: Apache-2.0
pragma solidity ^0.8.20;

/// @title ClprQueueMetadataXdr
/// @notice Pure XDR decoder for ClprQueueMetadata calldata.
///         Generated by PBJ — do not edit.
library ClprQueueMetadataXdr {

    // -----------------------------------------------------------------------
    // XDR wire-format field offsets (relative to the start of the struct in calldata)
    // Each singular uint64 field: 4-byte presence flag + 8-byte hyper
    // Each bytes[32] field:       4-byte presence flag + 32-byte opaque[32]
    // -----------------------------------------------------------------------
    uint256 private constant OFFSET_NEXT_MESSAGE_ID_PRESENCE      =   0;
    uint256 private constant OFFSET_NEXT_MESSAGE_ID               =   4;
    uint256 private constant OFFSET_ACKED_MESSAGE_ID_PRESENCE     =  12;
    uint256 private constant OFFSET_ACKED_MESSAGE_ID              =  16;
    uint256 private constant OFFSET_SENT_RUNNING_HASH_PRESENCE    =  24;
    uint256 private constant OFFSET_SENT_RUNNING_HASH             =  28;
    uint256 private constant OFFSET_RECEIVED_RUNNING_HASH_PRESENCE =  60;
    uint256 private constant OFFSET_RECEIVED_RUNNING_HASH         =  64;
    uint256 private constant OFFSET_RECEIVED_MESSAGE_ID_PRESENCE  =  96;
    uint256 private constant OFFSET_RECEIVED_MESSAGE_ID           = 100;

    /// @notice Total byte length of a fully-present ClprQueueMetadata.
    uint256 internal constant ENCODED_SIZE = 108;

    // -----------------------------------------------------------------------
    // Struct returned by decode()
    // -----------------------------------------------------------------------
    struct ClprQueueMetadata {
        uint64  nextMessageId;
        uint64  ackedMessageId;
        bytes32 sentRunningHash;
        bytes32 receivedRunningHash;
        uint64  receivedMessageId;
    }

    // -----------------------------------------------------------------------
    // Full struct decoder
    // -----------------------------------------------------------------------

    /// @notice Decode a ClprQueueMetadata from calldata at the given byte offset.
    /// @param data   The full calldata buffer passed in from the caller.
    /// @param offset Byte offset within `data` at which the struct starts.
    /// @return Decoded struct.
    /// @dev Reverts if any presence flag is not exactly 1 (strict canonical validation).
    function decode(bytes calldata data, uint256 offset)
        internal
        pure
        returns (ClprQueueMetadata memory result)
    {
        result.nextMessageId      = _readUint64(data, offset + OFFSET_NEXT_MESSAGE_ID,
                                                offset + OFFSET_NEXT_MESSAGE_ID_PRESENCE);
        result.ackedMessageId     = _readUint64(data, offset + OFFSET_ACKED_MESSAGE_ID,
                                                offset + OFFSET_ACKED_MESSAGE_ID_PRESENCE);
        result.sentRunningHash    = _readBytes32(data, offset + OFFSET_SENT_RUNNING_HASH,
                                                 offset + OFFSET_SENT_RUNNING_HASH_PRESENCE);
        result.receivedRunningHash = _readBytes32(data, offset + OFFSET_RECEIVED_RUNNING_HASH,
                                                  offset + OFFSET_RECEIVED_RUNNING_HASH_PRESENCE);
        result.receivedMessageId  = _readUint64(data, offset + OFFSET_RECEIVED_MESSAGE_ID,
                                                offset + OFFSET_RECEIVED_MESSAGE_ID_PRESENCE);
    }

    // -----------------------------------------------------------------------
    // Single-field accessors (use when only one field is needed — zero allocation)
    // -----------------------------------------------------------------------

    /// @notice Read next_message_id without decoding any other fields.
    function nextMessageId(bytes calldata data, uint256 offset) internal pure returns (uint64) {
        return _readUint64(data, offset + OFFSET_NEXT_MESSAGE_ID,
                           offset + OFFSET_NEXT_MESSAGE_ID_PRESENCE);
    }

    /// @notice Read acked_message_id without decoding any other fields.
    function ackedMessageId(bytes calldata data, uint256 offset) internal pure returns (uint64) {
        return _readUint64(data, offset + OFFSET_ACKED_MESSAGE_ID,
                           offset + OFFSET_ACKED_MESSAGE_ID_PRESENCE);
    }

    /// @notice Read sent_running_hash without decoding any other fields.
    function sentRunningHash(bytes calldata data, uint256 offset) internal pure returns (bytes32) {
        return _readBytes32(data, offset + OFFSET_SENT_RUNNING_HASH,
                            offset + OFFSET_SENT_RUNNING_HASH_PRESENCE);
    }

    /// @notice Read received_running_hash without decoding any other fields.
    function receivedRunningHash(bytes calldata data, uint256 offset) internal pure returns (bytes32) {
        return _readBytes32(data, offset + OFFSET_RECEIVED_RUNNING_HASH,
                            offset + OFFSET_RECEIVED_RUNNING_HASH_PRESENCE);
    }

    /// @notice Read received_message_id without decoding any other fields.
    function receivedMessageId(bytes calldata data, uint256 offset) internal pure returns (uint64) {
        return _readUint64(data, offset + OFFSET_RECEIVED_MESSAGE_ID,
                           offset + OFFSET_RECEIVED_MESSAGE_ID_PRESENCE);
    }

    // -----------------------------------------------------------------------
    // Internal helpers
    // -----------------------------------------------------------------------

    /// @dev Read a 4-byte XDR presence flag. Reverts if not exactly 0 or 1.
    function _readPresence(bytes calldata data, uint256 flagOffset)
        private
        pure
        returns (bool present)
    {
        uint256 pos = data.offset + flagOffset;
        uint32 flag;
        assembly {
            flag := shr(224, calldataload(pos))
        }
        require(flag <= 1, "XDR: invalid presence flag");
        return flag == 1;
    }

    /// @dev Read an 8-byte XDR hyper (uint64) at valueOffset, with presence check.
    function _readUint64(bytes calldata data, uint256 valueOffset, uint256 flagOffset)
        private
        pure
        returns (uint64 result)
    {
        require(_readPresence(data, flagOffset), "XDR: required field absent");
        // calldataload loads 32 bytes big-endian from the absolute calldata position.
        // The uint64 occupies the first 8 bytes of those 32 bytes.
        // Shift right by 192 bits (24 bytes) to isolate the top 8 bytes as a uint64.
        uint256 pos = data.offset + valueOffset;
        assembly {
            result := shr(192, calldataload(pos))
        }
    }

    /// @dev Read a 32-byte XDR opaque[32] (fixed-length, no length prefix), with presence check.
    function _readBytes32(bytes calldata data, uint256 valueOffset, uint256 flagOffset)
        private
        pure
        returns (bytes32 result)
    {
        require(_readPresence(data, flagOffset), "XDR: required field absent");
        // opaque[32] is exactly 32 bytes — fits perfectly in one calldataload with no shift.
        uint256 pos = data.offset + valueOffset;
        assembly {
            result := calldataload(pos)
        }
    }
}
```

### Calldata Arithmetic — Why `>> 192` for uint64

`calldataload(pos)` always loads exactly 32 bytes into a `uint256`, most-significant byte
first. An XDR `hyper` (uint64) occupies the first 8 bytes of those 32 bytes:

```
Byte in 256-bit word:  0   1   2   3   4   5   6   7   8  ...  31
                      [b7][b6][b5][b4][b3][b2][b1][b0][--]...[--]
                       ← uint64 value (8 bytes) →
```

Shifting right by `192` (= 256 − 64) drops the trailing 24 bytes, leaving the uint64 in the
low 64 bits of the uint256. No masking needed because `uint64(...)` truncates to 64 bits.

For a 4-byte presence flag (`uint32`), shift right by `224` (= 256 − 32) to isolate the
leading 4 bytes.

For `bytes32` / `opaque[32]`, no shift is needed — the 32 bytes fill the entire word exactly.

## 2. Library vs Contract Structure

### Decision: `library` with `internal pure` functions

PBJ generates a Solidity `library` for each message type. All functions are `internal pure`.

### Justification

| Option | Gas model | Inlining | Reuse |
|--------|-----------|----------|-------|
| `library` with `internal` functions | No `DELEGATECALL` — inlined at call site | Yes — Solidity compiler inlines `internal` library functions | `using ClprQueueMetadataXdr for bytes calldata;` |
| Separate contract with `internal` | Same as above | Yes | Requires inheritance or import |
| `library` with `public` functions | `DELEGATECALL` per call (~700 gas overhead) | No | Deployed once, shared |

`internal` library functions are **inlined by the Solidity compiler** (identical to `internal`
functions in the calling contract). There is zero `DELEGATECALL` overhead. The pattern:

```solidity
using ClprQueueMetadataXdr for bytes calldata;
// ...
uint64 id = data.nextMessageId(offset);
```

reads cleanly and compiles to the same bytecode as writing the `calldataload` expression
inline. `public` library functions would incur a `DELEGATECALL` (~700 gas) on every field
read, which would be catastrophic for the CLPR hot path.

A standalone contract with `internal` functions would work equally well for gas, but `library`
is idiomatic for stateless, pure utility code in Solidity and enables the `using ... for`
syntax. PBJ uses `library` for all generated XDR decoders.

### Deployment

Because all functions are `internal`, the library bytecode is **copied into the calling
contract** at compile time — there is no separate on-chain library deployment. This also means
the ABI encoder/decoder boundary is never crossed.

## 3. Calldata Pointer Arithmetic Patterns

The generator emits one of two patterns depending on whether field offsets are known at
compile time.

### Pattern A — Fixed-size field extraction (constant offset)

Used when the field's offset from the struct start is a compile-time constant (all fields in
a fully fixed-size struct, or all fields before the first variable-length field).

```solidity
// XDR hyper (uint64) at a known constant offset from the calldata start
// calldataload is a Yul built-in; it must be called inside assembly { } blocks.
// Step 1: compute the absolute calldata position in Solidity
// Step 2: use assembly to call calldataload and shr(192, ...) to isolate the top 8 bytes
uint256 pos = data.offset + FIELD_OFFSET;
uint64 fieldName;
assembly {
    fieldName := shr(192, calldataload(pos))
}

// XDR int / presence flag (uint32) at a known constant offset
// shr(224, ...) shifts away the trailing 28 bytes to isolate the top 4 bytes
pos = data.offset + PRESENCE_OFFSET;
uint32 presence;
assembly {
    presence := shr(224, calldataload(pos))
}

// XDR opaque[32] (bytes32) at a known constant offset — no shift required
pos = data.offset + HASH_OFFSET;
bytes32 hash;
assembly {
    hash := calldataload(pos)
}
```

**Math summary:**
- `calldataload(pos)` → 32-byte big-endian `uint256`
- `>> 192` → isolates bytes 0–7 (uint64 / XDR hyper)
- `>> 224` → isolates bytes 0–3 (uint32 / XDR int / presence flag)
- no shift → uses all 32 bytes (bytes32 / XDR opaque[32])
- `>> (256 - N*8)` → general rule for extracting leading N bytes

This is the cheapest possible EVM decode: each field costs exactly one `CALLDATALOAD` (3 gas)
plus one `SHR` (3 gas) plus one cast, for **~8–10 gas per field**.

### Pattern B — Variable-length field iteration (pointer-advancing)

Used for `repeated` fields and variable-length `opaque<>` / `string` fields where the offset
to each element depends on the sizes of preceding elements.

```solidity
uint256 ptr = offset;  // pointer starts at the beginning of the repeated field

// Read 4-byte XDR array count (uint32)
// calldataload is a Yul built-in and must be called inside assembly { } blocks.
uint32 count;
uint256 pos = data.offset + ptr;
assembly { count := shr(224, calldataload(pos)) }
ptr += 4;

for (uint256 i = 0; i < count; i++) {
    // Access element at ptr — exact pattern depends on element type:

    // Case A: fixed-size element (e.g., uint64 hyper = 12 bytes: 4 presence + 8 value)
    uint64 elem;
    pos = data.offset + ptr + 4;
    assembly { elem := shr(192, calldataload(pos)) }
    ptr += 12;  // advance by fixed element size

    // Case B: variable-length element (e.g., opaque<> with length prefix)
    uint32 len;
    pos = data.offset + ptr;
    assembly { len := shr(224, calldataload(pos)) }
    uint32 paddedLen = len + ((4 - (len % 4)) % 4);  // round up to 4-byte boundary
    // ... use element bytes at (ptr + 4) through (ptr + 4 + len - 1)
    ptr += 4 + paddedLen;  // advance past length prefix + padded data
}
```

For fixed-element repeated fields, the compiler can replace the loop with:
```solidity
uint256 end = ptr + uint256(count) * ELEMENT_SIZE;
while (ptr < end) {
    // process element at ptr
    ptr += ELEMENT_SIZE;
}
```

For hashing use cases (running hash chain), the entire repeated field bytes can be passed
directly to `keccak256` via `calldatacopy` into a memory scratch buffer without parsing
individual elements.

## 4. Fixed-Size vs Variable-Length Struct Differences

| Property | Fixed-size struct | Variable-length struct | Mixed struct |
|----------|-------------------|----------------------|--------------|
| **Offset computation** | All offsets are compile-time constants | Each field offset depends on the actual lengths of all preceding variable-length fields | Fields before the first variable-length field use constants; fields after use a runtime pointer |
| **Loops required** | None | Yes — for repeated fields and variable-length opaques | Only for the variable-length suffix |
| **`calldataload` count** | One per field | One per field + one per length prefix | One per fixed field + two per variable field |
| **Gas per field** | ~8–10 gas (1 CALLDATALOAD + 1 SHR) | ~16–20 gas (2 CALLDATALOAD + 2 SHR for length + value) | Mixed |
| **Total gas predictability** | Constant gas (no input-dependent branching) | O(N × element_gas) for repeated fields | Partly predictable |
| **Solidity pattern** | Constant offsets, no loops | Iterator with advancing `ptr` | Constant offsets then iterator |
| **Generator complexity** | Low | Medium | Medium |

### Fixed-size struct example (`ClprQueueMetadata`)

Every field offset is a manifest constant. The decoder is a sequence of `calldataload` +
shift expressions with no branches (except presence validation).

```
field 1 offset = 0  (known at compile time)
field 2 offset = 12 (known at compile time: field 1 presence 4B + field 1 value 8B)
field 3 offset = 24 (known at compile time: + 4B + 8B)
field 4 offset = 60 (known at compile time: + 4B + 32B)
field 5 offset = 96 (known at compile time: + 4B + 32B)
```

### Variable-length struct example (`ClprMessage` with a variable-length payload)

The offset to field N depends on the actual byte length of the variable-length field that
precedes it, which is only known at runtime:

```
field 1 offset = 0       (constant)
field 2 offset = ptr     (runtime: ptr = 4 + 4 + roundUp4(len1))
field 3 offset = ptr     (runtime: advances further after field 2)
```

The generator tracks which fields are fixed-size and which are variable-length during code
generation, and switches from constant-offset arithmetic to pointer-advancing arithmetic at
the boundary.

### Mixed struct

For a struct where the first K fields are fixed-size and the remaining fields are
variable-length, the generator emits constant-offset code for fields 1..K, then initializes
`ptr = FIXED_PREFIX_SIZE` and uses the iterator pattern for fields K+1..N.

## 5. Solidity Version Constraints and Compiler Settings

### Minimum Solidity Version

```solidity
pragma solidity ^0.8.20;
```

**Why ^0.8.20:**
- `0.8.0` — SafeMath built in, no overflow by default in unchecked blocks; required for safe arithmetic
- `0.8.4` — Custom errors (`error Foo()`); used for revert messages in presence validation
- `0.8.20` — `push0` opcode support (EIP-3855); reduces deployment cost slightly
- `0.8.24` — transient storage (not needed here but harmless to target a recent version)

A minimum of `0.8.4` would suffice for functionality; `^0.8.20` is recommended to stay close
to current toolchain versions used in CLPR smart contract development.

### `calldata` vs `memory`

| Use case | Recommended storage location |
|----------|------------------------------|
| Input buffer passed to `decode()` | `calldata` — free to read, no copy cost |
| Decoded struct returned from `decode()` | `memory` — must be constructed; unavoidable allocation |
| Single-field accessor return value | value type (uint64, bytes32) — stack, no allocation |
| Hashing a variable-length field | Pass `calldatacopy` to `keccak256` directly — avoids memory for large payloads |

All decoder parameters use `bytes calldata data` (not `bytes memory data`). Reading from
`calldata` is free — it does not trigger memory expansion. Reading from `memory` requires
the data to have been copied first (CALLDATACOPY + memory expansion cost).

**When `memory` is appropriate:** When the decoded struct must be stored in a local variable
or returned from a `public`/`external` function, Solidity requires the `memory` qualifier.
For `internal` functions, the compiler can sometimes keep struct fields on the stack if they
are small, but in practice the `memory` keyword is always needed for struct return values.

### Optimizer Settings

| Optimizer setting | Effect on generated XDR decoders |
|-------------------|-----------------------------------|
| `runs=200` (default) | Good balance between deployment size and per-call gas |
| `runs=1000000` | Optimizes aggressively for repeated calls; reduces per-call gas by ~5–15% at cost of larger bytecode; recommended for CLPR hot paths |
| `viaIR=true` | Enables Yul intermediate representation; allows more aggressive inlining and dead-code elimination; recommended alongside high `runs` values |

**Recommendation:** `optimizer.runs=1000000` with `viaIR=true` for CLPR verification
contracts. The CLPR service contract will be called frequently on mainnet; deployment cost
(paid once) is less important than per-call gas.

### `unchecked` Arithmetic

Offset pointer arithmetic (`ptr += elementSize`) can safely use `unchecked` blocks to save
gas, since:
1. XDR element sizes are bounded (validated count × bounded element size)
2. Overflow would require processing data larger than the maximum calldata size (~16 MB in
   practice, far below `uint256` overflow territory)

The generator emits `unchecked { ptr += elementSize; }` for pointer advances inside loops.

## 6. Presence Flag Handling in Solidity

### Background

XDR presence flags are 4-byte big-endian integers with valid values `0` (absent) or `1`
(present). Any other value is a canonical encoding violation (see xdr-design.md §5.12).

### Two Strategies

**Strategy A — Strict validation (recommended for CLPR):**
```solidity
uint256 pos = data.offset + flagOffset;
uint32 flag;
assembly { flag := shr(224, calldataload(pos)) }     // 6 gas: CALLDATALOAD + SHR
require(flag <= 1, "XDR: invalid presence flag");   // ~13 gas: ISZERO 3 + JUMPI 10
if (flag == 0) revert FieldAbsent();                 // revert if absent when required
```

**Strategy B — Skip validation (gas-optimized, assumes canonical input):**
```solidity
// Trust that presence=1 for all required fields; read value unconditionally
uint256 pos = data.offset + valueOffset;
uint64 value;
assembly { value := shr(192, calldataload(pos)) }
// (presence flag bytes are read implicitly as part of the offset arithmetic)
```

### Tradeoff Analysis

| Factor | Strict validation | Skip validation |
|--------|-------------------|-----------------|
| Gas per field | +19 gas (CALLDATALOAD 3 + SHR 3 + REQUIRE 13 per field) | 0 extra gas |
| Security | Rejects malformed / non-canonical input | Silently accepts corrupt data |
| CLPR use case | Calldata originates from untrusted off-chain submitters | N/A |

### Recommendation

For CLPR, **use strict validation**. Calldata is submitted by external parties; a malformed
presence flag (e.g., `0xDEADBEEF`) could cause silent misinterpretation of field boundaries.
The cost is ~19 gas per field × 5 fields = ~95 gas for `ClprQueueMetadata` — negligible
compared to the overall verification cost.

For internal dev/test environments where input is known to be canonical, skip validation to
reduce noise in gas benchmarks.

The generator should emit a configurable option (e.g., a `bool strictPresence` compile-time
parameter) so the calling project can choose, with strict mode as the default.

## 7. Gas Cost Analysis

### Opcode Reference Costs (EVM Yellow Paper / EIP-3529)

| Opcode | Gas cost |
|--------|----------|
| `CALLDATALOAD` | 3 |
| `SHR` | 3 |
| `AND` | 3 |
| `ADD` | 3 |
| `MSTORE` | 3 (warm) |
| `MLOAD` | 3 (warm) |
| `REQUIRE` (revert not taken) | ~13 (ISZERO 3 + JUMPI 10) |

### ClprQueueMetadata — Opcode Breakdown

All 5 fields present, strict presence validation enabled. The decoder reads each presence flag
then the value.

| Field | Presence read | Value read | Ops | Gas |
|-------|--------------|-----------|-----|-----|
| `next_message_id` presence | CALLDATALOAD + SHR(224) + REQUIRE | — | 3 ops | 19 |
| `next_message_id` value | CALLDATALOAD + SHR(192) + MSTORE | — | 3 ops | 9 |
| `acked_message_id` presence | CALLDATALOAD + SHR(224) + REQUIRE | — | 3 ops | 19 |
| `acked_message_id` value | CALLDATALOAD + SHR(192) + MSTORE | — | 3 ops | 9 |
| `sent_running_hash` presence | CALLDATALOAD + SHR(224) + REQUIRE | — | 3 ops | 19 |
| `sent_running_hash` value | CALLDATALOAD (no shift) + MSTORE | — | 2 ops | 6 |
| `received_running_hash` presence | CALLDATALOAD + SHR(224) + REQUIRE | — | 3 ops | 19 |
| `received_running_hash` value | CALLDATALOAD (no shift) + MSTORE | — | 2 ops | 6 |
| `received_message_id` presence | CALLDATALOAD + SHR(224) + REQUIRE | — | 3 ops | 19 |
| `received_message_id` value | CALLDATALOAD + SHR(192) + MSTORE | — | 3 ops | 9 |
| Memory allocation for struct | MSTORE × 5 fields (already counted above) | — | — | 0 |
| **Total** | | | | **~134 gas** |

Note: memory expansion for the returned struct costs an additional ~12–20 gas depending on
prior memory usage (first-time expansion of 5 × 32-byte slots = 160 bytes).

**Design doc §8.2 estimate of ~40 gas** corresponds to the no-validation path where only the
5 value reads are counted (5 × CALLDATALOAD at 3 gas = 15 gas) plus shifts (5 × 3 = 15 gas) =
30 gas, plus small overhead ≈ 40 gas. The 134 gas estimate above includes strict presence
validation; the 40 gas estimate is the skip-validation lower bound.

### ClprMessagePayload Discriminant Read

The discriminant is a 4-byte XDR int at the start of the union:

```solidity
// One CALLDATALOAD + one SHR(224) = 6 gas (calldataload is a Yul built-in, requires assembly)
uint256 pos = data.offset + ptr;
uint32 discriminant;
assembly { discriminant := shr(224, calldataload(pos)) }
ptr += 4;
```

With bounds check (`require(discriminant <= MAX_DISCRIMINANT, ...)`): +13 gas = **~19 gas total**
(corrected from the §8.2 ~8 gas estimate, which used JUMPI=3; the correct JUMPI cost is 10 gas).

### Repeated Message Iteration

For a bundle of N messages where each message has a fixed XDR encoded size S:

```
4 gas  — CALLDATALOAD + SHR for count read (once)
per iteration:
  3 gas — ADD for ptr advance
  S/4 × (3 + 3) gas — reads for each 8-byte field within element (if fully decoded)
```

If the repeated field is processed only for hashing (not decoded field by field):

```
3 gas  — count read (CALLDATALOAD + SHR)
4 gas  — compute total byte length (MUL + ADD)
6 gas  — CALLDATACOPY to memory scratch area
30 gas — KECCAK256 for first 32 bytes (6 gas + 1 gas per 32-byte word)
```

For a bundle of N messages, hashing the entire array without field-level decoding:
`~13 + N × 1` gas for the KECCAK256 word cost, far cheaper than decoding each field.

### Total Gas Estimate for CLPR Bundle Verification

Assuming: 1 `ClprQueueMetadata` + N messages in the bundle, presence validation on:

| Step | Gas |
|------|-----|
| Decode `ClprQueueMetadata` | ~134 gas |
| Read message array count | ~8 gas |
| Read each message discriminant (N messages) | N × ~19 gas |
| Reconstruct running hash chain (keccak per message) | N × ~50 gas |
| Validate final running hash against expected | ~30 gas |
| **Total** | **~172 + N × 69 gas** |

For a typical bundle of 100 messages: ~7,072 gas. Negligible compared to the ECDSA/BLS
signature verification that typically dominates CLPR gas costs.

## 8. Rejected Alternatives

### ABI-encoded calldata vs XDR

Standard Solidity ABI encoding uses 32-byte slots for all types, length-prefixed dynamic
arrays, and a separate head/tail layout for dynamic data. For a `uint64` field, ABI encoding
uses 32 bytes (24 wasted zero bytes); XDR uses 12 bytes (4 presence + 8 value). The XDR
wire is ~3× smaller for integer-heavy messages.

More critically, ABI encoding is not suitable for cross-chain interoperability. ABI is
Ethereum-specific; XDR is an IETF standard (RFC 4506) used in NFS, Stellar, and other
systems. CLPR requires a format that can be encoded by the Java node and decoded by the
EVM contract with minimal glue code. ABI would require a custom Java ABI encoder, whereas
XDR encoding is handled natively by PBJ's Java XDR codec.

**Rejected because:** Larger wire format, Ethereum-specific (not cross-chain), requires
extra Java-side encoder complexity.

### `calldatacopy` to memory then reading vs direct `calldataload`

`calldatacopy(destMem, srcCalldata, len)` copies bytes from calldata to memory, after which
fields can be read with `mload`. The cost is 3 gas per 32-byte word copied plus memory
expansion costs. For a 108-byte `ClprQueueMetadata`: 4 × 3 = 12 gas for copy + memory
expansion (~15 gas) = ~27 gas just to set up, before any field is read.

Direct `calldataload` reads are 3 gas each, no setup cost. For 5 fields: 5 × 3 = 15 gas
total, cheaper even without accounting for the copy overhead.

`calldatacopy` to memory is only appropriate when a large contiguous block of data must be
passed to a hash function (`keccak256` requires a memory pointer) or when the same bytes
will be read many times (>3× per word). For the CLPR decoder, each field is read at most
once, so direct `calldataload` is always preferred.

**Rejected because:** Higher setup cost for one-time field reads; memory expansion is
quadratic above 724 bytes; direct `calldataload` is simpler and cheaper.

### Generating Yul assembly vs Solidity with inline `assembly { }` blocks

Yul (EVM assembly with structured control flow) allows manual control over every opcode. In
principle, a Yul decoder could shave a few gas by eliminating Solidity's ABI encode/decode
wrapper for the return struct, reusing registers, and avoiding stack shuffling.

In practice, Solidity with `viaIR=true` and `runs=1000000` generates Yul internally and
applies the same optimizations. The generated Yul from Solidity IR is typically within 1–3%
of hand-written Yul for simple, pure, calldata-reading code. Hand-written Yul is harder to
audit, harder to maintain, and not supported by common Solidity toolchains (Hardhat, Foundry
test infrastructure, slither analysis).

`calldataload` is a Yul built-in accessible via Solidity's `assembly { }` blocks. The inline
assembly pattern `result := shr(192, calldataload(pos))` compiles to exactly two opcodes
(`CALLDATALOAD`, `SHR`) with no additional overhead. Generating a full Yul contract rather
than Solidity with inline assembly blocks adds no further gas benefit for the decoder use case.

**Rejected because:** Marginal gas savings (≤3%) do not justify loss of auditability,
toolchain compatibility, and maintainability. Solidity with IR optimizer achieves equivalent
output.

## Design Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Library vs contract | `library` with `internal` functions | Compiler inlines at call site; zero DELEGATECALL overhead; idiomatic for stateless pure decoders |
| Solidity version | `^0.8.20` | Custom errors, unchecked arithmetic, push0 opcode support |
| Optimizer | `runs=1000000`, `viaIR=true` | CLPR hot path; per-call gas dominates deployment gas |
| Presence validation | Strict by default (configurable) | Calldata from untrusted submitters; malformed presence flags could corrupt field boundary math |
| `calldata` vs `memory` | `calldata` for all inputs, `memory` for decoded struct | No copy cost for input; struct must be in memory for return |
| Field access strategy | Inline assembly `calldataload` | Cheaper than copy-to-memory; Yul built-in requires `assembly { }` blocks |
| Yul vs Solidity | Solidity with IR | Equivalent gas, better auditability |
